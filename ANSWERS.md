### Ответы на часть 3 ###

**3.1.** В базу попадёт статус, переданный в запросе, который будет обработан вторым, при этом, в моей реализации, в истории обе записи со сменой статуса будут иметь начальный статус `in_progress`. Для решения можно добавить поле для отслеживания версии в задачу, вроде `version: int = Field(default=1, nullable=False)`. При обновлении статуса фильтровать из базы задачу по версии и id, и, если такой уже не существует, то отдавать ошибку 409. Если запись прошла нормально, то увеличивать версию на 1

**3.2** Проверю для начала в базе, как выглядит задача 4521, если она действительно в статусе "review", посмотрю историю изменения статусов. Если в истории статус изменился, то проверю что у нас с логикой смены статуса у самой задачи, почему он не меняется, хотя запрос пришёл и в истории зафиксирован, может откатилась запись в базу из за ошибки? Если нет записи в истории, гляну логи обращения к серверу за это время, проверить что запрос вообще до нас дошёл, посмотрю коды ответов. Дальше по ситуации буду на локалке или тестовом сервере прогонять сценарий и пытаться воспроизвести баг.

**3.3** Коментарий к смене статуса я уже добавил в этот проект, только позже проверил по ТЗ что вообще-то это не нужно. Можно сделать поле в БД и валидаторах обязательным. Для комментариев к задачам сделаю новую модель, свяжу её м-1 к задачам, чтоб можно было оставить несколько коментов для задачи. Также привяжу её к пользователю через м-1 для отслеживания авторства. Кроме поля текста, ещё добавлю время создания.

**3.4** Начну проверку с БД, через EXPLAIN ANALYZE для таблицы задач. Можно добавить составной индекс для частой комбинации фильтров для задач, вроде полей статуса и id проекта задачи. Можно добавить механизм чистки старых, выполненных задач через Celery, чтоб база не росла бесконечно, если такое не противоречит ТЗ. Может быть использовать keyset pagination, заместо offset.
